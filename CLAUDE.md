# CLAUDE.md — developing wordguess

Guidance for anyone (human or Claude) writing code in this repo. It exists to keep
two goals in balance:

1. **A showcase pet project** — clean, coherent, something you'd be happy to link on
   a CV. That means real engineering hygiene: small PRs, tests, readable diffs.
2. **A learning vehicle** — the point is to *understand* how a tiny LLM works, one
   token at a time. So code is written to be read and reasoned about, not to be
   maximally clever or generic.

When those two goals ever tension, **learning wins**. A slightly longer, explicit
implementation that the owner can read in one sitting beats a compact abstraction
that hides the mechanics.

---

## What this project is (one paragraph)

A web game where you try to out-predict a small char-level GPT. You read a slice of a
book the model was trained on and guess which next *word* the model finds most likely,
picked from candidates the model itself generated by greedy rollout. Your score is the
probability mass your guess carried. See [README.md](README.md) for the full mechanic
and API contract.

---

## Repo layout

```
model/     char-level GPT definition, tokenizer, training script
server/    FastAPI app: puzzle generation, rollout, scoring, in-memory cache
web/       single-page frontend (vanilla JS, no build step)
corpora/   raw training texts + trained checkpoints (weights gitignored)
tests/     pytest tests, mirroring the package they cover (test_tokenizer.py, …)
scripts/   dev + tooling scripts
```

**Tests live in the top-level `tests/` directory**, run with `pytest`, one file per
unit under test (`tests/test_tokenizer.py`, `tests/test_model.py`,
`tests/test_puzzle.py`, …). Keep the tree flat and mirror the module names so a test
is easy to find. The lightweight runnable `__main__` demos described below stay inside
their own module (they're sanity checks, not the test suite).

`model/` and `server/` are Python packages. `corpora/<name>/` holds one corpus's
`input.txt` plus its trained checkpoint (the checkpoint is **not** committed).

## Development environment

CPU-only is a hard requirement — everything must run on a laptop with no GPU.

```bash
cd ~/dev/wordguess
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt        # torch, fastapi, uvicorn (see M1 #1)
```

`torch` is the one heavy dependency (hundreds of MB). Install it once into the venv;
never add it to anything that must stay lightweight. Keep the dependency list short —
if you're reaching for a big library, stop and ask whether a dozen lines of plain code
would teach more and cost less.

### Running things

```bash
# Train a model for one corpus (writes a checkpoint under corpora/<name>/)
python -m model.train --corpus corpora/<name>/input.txt

# Serve the game
uvicorn server.app:app --reload
# then open web/ (served by the app) in a browser

# Health check
curl localhost:8000/health   # -> {"status": "ok"}
```

Exact commands are pinned by each issue's acceptance criteria — treat those as the
contract.

## How we work: backlog-driven, one issue at a time

**GitHub is the single source of truth for the backlog** — there is no backlog file
in the repo. Work is tracked entirely as GitHub Issues and on the project board:

- **Issues** — <https://github.com/stefanosello/wordguess/issues>, across four
  milestones (M1–MVP → M4–Polish).
- **Project board** — <https://github.com/users/stefanosello/projects/3> (Status +
  Milestone fields).
- Read and edit issues with the `gh` CLI (`gh issue list`, `gh issue view <N>`,
  `gh issue edit <N>`) or in the web UI. Don't reintroduce a generated backlog file.

**Milestone order matters and is intentional:**

- **M1: MVP** — prove the whole loop end to end with a deliberately simple
  *single-token* puzzle (correct = argmax next char). Skeleton → model → tokenizer →
  training → inference → `/puzzle` + `/guess` → minimal frontend.
- **M2: Next-word puzzles** — upgrade the mechanic to the real thing: greedy
  rollout to whole words + probability-mass scoring.
- **M3: Multiple corpora** — train/serve several books, let the player choose.
- **M4: Polish & deploy** — session streaks, a design pass, run docs.

After M1, **stop and confirm the loop works end to end** before starting M2.

### Per-issue loop

1. Read the issue's **scope**, **acceptance criteria**, and **out-of-scope** notes.
   The out-of-scope section is load-bearing — it exists to stop scope creep. Ship the
   thin thing, then iterate.
2. Branch, using the naming convention below: `git switch -c feature/m1-model-def`.
3. Implement **only** what's in scope.
4. Meet **every** acceptance checkbox before calling it done.
5. Open a PR that closes the issue (`Closes #N` in the body).

### Git / PR conventions

- **Never commit directly to `main`.** `main` only ever advances through a merged PR.
  Always start work on a dedicated branch, push it, and open a PR — even for a one-line
  docs fix. Treat `main` as read-only in your working copy. (A GitHub ruleset enforces
  this server-side.)
- **Branch names are `<type>/<short-name>`**, where `<type>` describes the task:
  - `feature/` — a new capability (e.g. `feature/m1-model-def`)
  - `bugfix/`  — a fix (e.g. `bugfix/puzzle-cache-expiry`)
  - `doc/`     — documentation (e.g. `doc/readme-setup`)
  - `infra/`   — tooling, CI, deployment (e.g. `infra/dockerfile`)

  Keep `<short-name>` lowercase and hyphenated, and prefix it with the milestone when
  relevant (`m1-`, `m2-`, …).
- Small, focused branches — one issue per PR. A reviewer should grasp the diff in a
  few minutes.
- Present-tense, imperative commit subjects ("Add char-level tokenizer"), a short body
  explaining the *why* when it isn't obvious.
- Keep PRs green: the acceptance criteria are your definition of done.
- Never commit model weights — `.gitignore` already blocks `*.pt`/`*.bin`/checkpoints.
  If `git status` ever shows a weight file, something is wrong.

## Coding conventions (readability first)

- **Write for a reader learning LLMs.** Prefer explicit tensors and named steps over
  dense one-liners. A comment that says *why* (or what a tensor's shape means) is worth
  more than clever code.
- **No premature abstraction.** Don't add a base class, a plugin system, or a config
  framework until there are two real cases that need it. Duplication is cheaper than the
  wrong abstraction here.
- **Small functions with honest names.** `predict_next(prompt) -> list[(char, prob)]`
  tells you everything. Keep signatures boring.
- **Type hints and short docstrings** on public functions. Skip ceremony on obvious
  internals.
- **Comment tensor shapes** at every non-trivial step, e.g. `# (batch, block_size, n_embd)`.
  In a transformer, shape confusion is the #1 source of bugs and misunderstanding.
- **Fail loudly.** Bad puzzle_id → clean 404; out-of-range token id → raise, don't
  silently corrupt. Silent wrongness is the enemy of learning.

### Testing

- Every non-trivial primitive gets a tiny, fast test or a runnable `__main__` demo:
  tokenizer round-trip, a forward-pass shape check, `predict_next` sums to ~1,
  a rollout that prints candidate words + probabilities.
- Tests go in the top-level `tests/` directory (`pytest`), one file per unit —
  see [Repo layout](#repo-layout). Runnable `__main__` demos stay in their module.
- Tests should run on CPU in seconds. Use a tiny model config in tests, not the real one.
- You don't need 100% coverage — you need enough to trust the mechanic and to *show*
  the mechanic working.

## Invariants that must never break

These are the load-bearing rules of the game and the project. Treat a violation as a bug:

1. **The correct answer is never inferable from `POST /puzzle`.** Choices are shuffled
   and the answer + probabilities live only in the server-side cache, revealed solely by
   `POST /guess`. If a frontend or response ever leaks the answer early, the game is
   broken.
2. **Model weights are never committed.** They're trained locally from corpora.
3. **Everything runs on CPU** with small defaults (n_layer≈4, n_head≈4, n_embd≈128,
   block_size≈128). Megabytes, not gigabytes.
4. **One model per corpus, loaded once** at startup and reused across requests — never
   reloaded per call.
5. **Corpora must be public domain** (e.g. Project Gutenberg) if committed as text.

## The learning thread (why the code is shaped this way)

This project is a guided tour of how a language model predicts text. Each milestone
teaches one idea — keep the code faithful to the concept so the owner can connect code
to intuition:

- **Tokenizer (M1)** — the model sees integers, not characters. `encode`/`decode` is the
  bridge. Char-level keeps the vocabulary tiny and the mapping trivial to inspect.
- **The GPT (M1)** — embeddings turn token ids into vectors; positional embeddings add
  order; causal self-attention lets each position look only *backward*; a linear head
  turns the final vectors into a score per vocabulary character. Comment the shapes and
  the causal mask — that's where the "aha" lives.
- **Next-char distribution (M1)** — softmax over the head's logits *is* the model's
  belief about what comes next. The single-token puzzle exposes this directly: the
  argmax is the answer, ranks 2..K are the plausible distractors.
- **Greedy rollout to words (M2)** — a "word" is several tokens. Rolling forward
  token-by-token and multiplying probabilities shows how confidence compounds (and
  decays) across a sequence. This is the core intuition the game is built to teach.
- **Probability-mass scoring (M2)** — rewards being *close* in the model's own belief,
  not just right/wrong. It makes the probability distribution tangible to the player.

When you implement a milestone, add a couple of sentences to the README or a
`docs/`/`NOTES.md` explaining the concept in plain language. Teaching it back is part of
the deliverable.
